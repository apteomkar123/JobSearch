SESSION 9 — FETCH JD + ATS SCORE FIXES
════════════════════════════════════════════════════════════

1. FIXED FETCH JD SKIPPING MOST JOBS
   File: js/api.js — fetchAllJDs and autoFetchJD
   Root cause: skip logic used `if (existing !== undefined)` which permanently blocked
   any job previously marked accessible:false (transient failures, network errors, etc.).
   Fix: changed skip condition to only skip entries that are confirmed accessible=true
   AND have a computed atsScore. Previously-failed jobs are now retried on every run.

2. FIXED ATS SCORES BEING ARTIFICIALLY LOW
   File: js/api.js — computeATSFromJD
   Root cause: formula was `matches / total_unique_jd_words * 100`. A JD has ~300
   unique non-stopword words; even a great resume match covers only ~80, giving 27%.
   Fix: changed to frequency-filtered scoring — only words appearing 2+ times in the JD
   (actual skill requirements, not boilerplate) form the denominator. Added tag-coverage
   component (words from extracted job.tags matched to resume vocab). Weighted blend:
   tagScore * 0.55 + keyScore * 0.45 → produces 75-95% for well-matched candidates.

3. FIXED needsBoost NEVER FIRING FOR JD-BUILT RESUMES
   File: js/api.js — fetchAllJDs, autoFetchJD, parseJD
   Root cause: buildAndStoreResume is called before atsScore is computed, so
   calcATSScore inside buildResumePDF returned null, making needsBoost always false.
   The keyword-stuffing sections (EHS skills, Data/Automation, GIS, AI/Technical)
   and extra bullet fill-up never activated for JD-built resumes.
   Fix: pre-seed window.RESUMES[id] = {atsScore: 0, freshBuild: true} before calling
   buildAndStoreResume so needsBoost = (0 < 85) = true. The real atsScore is written
   back after the build completes, overwriting the placeholder.

9. FIX — RENDER BUG: TypeError: Cannot read properties of undefined (reading 'map') (2026-06-08)
   The async resume-chunk loader IIFE in state.js called window.render() after
   completing (success or failure). If the render function threw synchronously,
   it caused an uncaught promise rejection.
   Additionally, window.ALL_JOBS could theoretically contain null/undefined
   elements (from corrupted localStorage or a failed push), causing .map(j=>j.id)
   calls to throw "Cannot read properties of undefined".
   Fixes applied:
   - state.js: wrapped final window.render() call in try-catch to prevent
     uncaught promise rejection.
   - ui.js render(): added Array.isArray guard + .filter(Boolean) on ALL_JOBS
     so null/undefined elements never reach .map() or .filter() calls.
   - ui.js updateStats(): same .filter(Boolean) guard added.
   - ui.js parseBulkUrls(): same .filter(Boolean) guard before Math.max(ids).
   - js/api.js parseJobUrl(): same .filter(Boolean) guard before Math.max(ids).

11. ATS SCORE ACCURACY OVERHAUL + RESUME AUTO-BOOST (2026-06-08 session 2)

    Problem: Prior ATS calculator used substring matching (tag.includes(skill))
    which caused gross false positives:
      - 'ai'.includes check matched 'training' (tr-AI-ning contains 'ai')
      - 'bi'.includes check matched 'ability', 'combine', etc.
      - 'r'.includes check matched almost every word
    Result: scores were inflated and meaningless.

    Fix — Accurate word-boundary ATS calculator (js/ui.js):
    - Each job tag is split into individual words on spaces/hyphens/slashes
    - Each word is checked against ATS_SKILL_WORDS (a Set for O(1) lookup)
    - A tag "matches" if ANY of its words is in the candidate's verified skill set
    - Stopwords ('and','for','the', etc.) are excluded from word matching
    - Short skills ('bi','ai','gis','r') are in the Set but only match as WHOLE WORDS
      (Set.has() is exact — 'bi' will not match 'ability', only the word 'bi' itself)
    - window.calcATSScore exposed globally so resume.js can access it

    Resume auto-boost (js/resume.js):
    - buildResumePDF() now calls window.calcATSScore(job) at the start
    - If score < 85, needsBoost = true
    - When needsBoost: missing high-value GP bullets are appended to selectedBullets
      (capped at 8 total to preserve 2-page constraint)
    - The 3 standard skill lines are always shown regardless of job type,
      so keyword coverage is already comprehensive
